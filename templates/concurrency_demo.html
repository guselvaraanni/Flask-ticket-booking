Why this project exists

Two users clicking “Book” on the same seat at the same millisecond must not both succeed. Here is exactly how we prevent that.

SELECT ... FOR UPDATE

PostgreSQL acquires an exclusive row lock on the seat before checking availability.

seat = Seat.query.with_for_update().filter_by(id=seat_id).first()

Unique constraint

bookings.seat_id is unique — a second insert fails even if application logic regressed.

ThreadPoolExecutor

tests/simulate_load.py fires 51 concurrent HTTP requests. Exactly one returns 201.

Concurrent booking flow

Request 1 (User A)

POST /api/bookings

↓
Lock acquired
↓
status = BOOKED + INSERT booking
↓
COMMIT → 201 Created
same seat

Request 2 (User B)

POST /api/bookings
↓
WAIT for lock...
↓
Lock acquired — reads BOOKED
↓
409 Conflict
